## Round 0（2026-07-23）— red-team

**検証根拠（事実のみ）**

- Live: https://paymap-six.vercel.app — ヘッダ「使えるペイ · 渋谷MVP / 16件」
- シード: `data/shops.json` 16件・架空店名（「北口中華そば」等）・README 明示のサンプル座標
- `/api/reports` → `{"stats":[],"source":"supabase"}`（報告ゼロ）
- リポジトリ内に福岡・天神・博多・ISSIN・実証・事業計画・KPI・法人の記述なし（本ログ以外）
- DB: Loomap 同居（README）。専用主体なし

### 採点

| # | 軸 | 点 | Why |

### 総合・判定

| 項目 | 結果 |
|---|---|
| **総合点** | **21 / 100** |
| **採択判定** | **ほぼ否** |

FGN / ISSIN / 実証フルサポート系で「福岡で回す実証」を選ぶ場では、現状は技術プロトタイプの域を出ず、地域・データ・体制・資料の四点で門前払い級。

### Top 5 blockers（採択インパクト順）

1. **福岡シードがゼロ**（渋谷 MVP）— 地域適合性を即死させる
2. **サンプル／架空店舗の自認** — データ信頼性とデモ信用を破壊
3. **報告トラクションゼロ**（`stats: []`）— クラウドソース仮説が未検証
4. **市向け資料・実証ストーリー不在** — 政策接続を語れない
5. **実施主体・収益・体制の欠落** — 実証フルサポートの「誰が責任を持つか」に答えられない

### Top 5 concrete repo improvements（次スプリント・コード/docs で実装可）

1. **`data/shops-fukuoka.json` + `/area/fukuoka`（天神/博多）** — 実在店または OSM 由来の明示ソース付きシードに差し替え／追加。ヘッダから「渋谷MVP」を福岡デモ時は外す
2. **架空渋谷シードの扱いを明示分離** — `sample: true` / UI で「デモ用サンプル」バッジ、または福岡実データのみをデフォルト地図にする（偽店を実在に見せない）
3. **`docs/pitch-fukuoka.md`** — 市 DX・キャッシュレス・観光インバウンドへの接続、実証仮説、対象エリア、成功 KPI（例: 報告数・フィルタ利用率）を1本の日本語ストーリーで固定
4. **デモ用ランディング or `/for-city`** — 審査向けに課題→地図→報告1タップ→信頼度更新の導線。英語1文で inbound を明示
5. **シード出典・最終確認日・報告数を UI/JSON に露出** — `source`/`lastVerified`/`reportCounts` を PinSheet と README に載せ、空 stats でも「検証プロセス」を見せる（虚偽トラクションは禁止）

### ≥90 になるために真である必要があること（ハード要件）

1. **福岡パイロットエリアに実在店舗シードが審査に耐える密度**（サンプルではない。出典・更新日付き）
2. **実ユーザー報告または店舗確認の非ゼロ・再現可能なトラクション**（デモ捏造不可）
3. **市／観光・商工・キャッシュレス関連の実証フレーム**（対象・期間・KPI・協力先）が文書化され製品デモと一致
4. **実施主体が特定できる**（個人でも可だが責任者・運用・個人情報/報告データの扱いが明確）
5. **競合に対する楔が実測で示せる**（例: Google に無い「失敗報告で落ちる決済」が現地で再現）
6. **ピッチ資料が存在し、地図デモが福岡で30秒以内に理解される**
7. **DB/運用が「トイレ地図の同居無料枠」依存を審査説明可能な状態に整理**（専用 or 明示的一時措置＋移行計画）
