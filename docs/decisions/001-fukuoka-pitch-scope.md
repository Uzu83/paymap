# ADR 001: 福岡ピッチ本線のスコープ

- **Status**: Accepted
- **Date**: 2026-07-23
- **Context**: Paymap を福岡市向け支援プログラム（FGN / 実証実験フルサポート / 県 ISSIN 等）に提出するため、プロダクト・データ・インフラの「本線」をどこに置くかを決める必要があった。

## 決定

1. **デフォルトエリアは福岡（天神〜博多）** とする。渋谷は `sample: true` の比較用デモに降格する。
2. **店舗位置のシードは OpenStreetMap（OSM）実在 POI** を使う。座標・名称は OSM 由来、決済対応は別レイヤーで管理する。
3. **決済シードに `field`（現地確認済）を捏造しない**。初期は `estimated`（チェーン一般推定）と `unverified`（未検証）のみ。現地確認・ユーザー報告で段階的に昇格する（`docs/payment-status-promotion.md`）。
4. **報告 DB は Loomap（公衆トイレ地図）の Supabase 無料枠に一時同居** する。パイロット採択または報告安定・枠逼迫で Paymap 専用プロジェクトへ分離する。

## 根拠

### なぜ福岡デフォルトか

- 市 DX 戦略にキャッシュレス推進が明示されており、説明の接点が明確。
- 観光・インバウンドの「飲食前の決済不安」は繰り返し指摘される課題で、地図＋報告の仮説と一致する。
- 姉妹プロダクト Loomap で福岡 OSM シード運用の知見があり、エリア選定コストが低い。
- 渋谷は初期 MVP の架空・サンプル座標が残っており、審査本線として誠実さに欠ける。比較デモとして分離表示する方が信頼できる。

### なぜ OSM か

- 実在 POI の座標・名称をライセンス明確（ODbL）に取得でき、ピッチで「実在店ベース」と説明できる。
- 福岡エリアの POI 密度は手作り座標より拡張しやすく、カバー率 KPI と整合する。
- 決済手段は OSM に十分な鮮度がないため、**位置は OSM・決済は別シード＋報告** の二層に分ける。

### なぜ fake `field` ステータスを出さないか

- 審査・市担当者向けピッチで「現地確認済」と嘘をつくと、実証開始時の信頼毀損が不可逆。
- UI バッジ（チェーン推定 / 未検証）と採点ログ（`docs/pitch-fukuoka-score.md`）で、**正直な未完了状態** を示す方が長期的に採点・パートナー信頼に効く。
- red-team 評価でも code-only の天井は ~64 点であり、捏造トラクションはリスクだけが増える。

### なぜ Loomap DB 同居（一時）か

- 個人開発・無料枠制約下で、新規 Supabase プロジェクトの枠が空くまで報告 API を動かす必要がある。
- テーブル・RLS を Paymap 用に分離し、採択・報告安定後にエクスポート移行する条件を `/for-city` と本 ADR で明示する。

## 結果

- 地図デフォルト・`/for-city`・`/area/fukuoka` が福岡本線で一貫する。
- シード ~68 店（2026-07 時点）: チェーン推定 13 / 未検証 55。報告はゼロから積み上げる。
- 渋谷・サンプル店は sitemap に含めるが優先度を下げ、審査説明からは外す。

## 見直しトリガー

- パイロット採択 → DB 分離タスクを起票
- 週次実報告が安定（目安 50 件以上）→ `field` 昇格オペの本格化
- 福岡以外の自治体パイロット打診 → ADR 002 でエリア戦略を再検討
